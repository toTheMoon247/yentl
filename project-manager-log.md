## Day 2 — 2026-05-31 → 2026-06-01

**Today.** Cleared the CI blocker, validated the Matchmaker role gate, finished the auth system's session/logout work, and set up milestone-based versioning. The project now has its first known-good rollback point: **`v0.1.0 — Authentication`**.

CI — fixed and green:

- Root cause of the iOS build failures was *not* the Xcode version. The jobs built with `xcodebuild -target`, which doesn't construct the transitive SwiftPM product-dependency graph — so `swift-clocks` couldn't see `ConcurrencyExtras` (`no such module 'ConcurrencyExtras'`). The earlier guesses (Xcode version, `git init.defaultBranch`, `SWIFT_ENABLE_EXPLICIT_MODULES=NO`) were chasing the wrong cause.
- Fix: added **shared schemes** for both apps (they only existed as autogenerated user schemes, so weren't committed) and switched CI to `-scheme`. Dropped both dead workarounds. All four jobs (shared package, both iOS builds, SwiftLint) now green.

Phase 1 — Matchmaker role gate validated:

- Signed into the Matchmaker app via Google → `user` role correctly showed **Access Pending** → promoted role in Studio → signed in again → routed to `MatchmakerHomeView`. The last unverified piece of the core auth flow.

Phase 1 — session persistence + logout:

- Found that both already worked via Supabase SDK defaults (Keychain storage + auto-refresh; `signOut()` clears the local session before the network revoke, so logout is reliable even offline).
- Made it deliberate rather than accidental: pinned the Keychain/auto-refresh options explicitly in `Backend.swift` (guards against a future SDK default change) and enabled `emitLocalSessionAsInitialSession` for faster, offline-capable signed-in restoration on cold launch.
- DRY'd the three duplicated sign-out buttons into a shared `SignOutButton` in `YentlShared`.

Hiccup worth remembering:

- Local Xcode build of YentlMatchmaker failed with "Missing package product 'YentlShared'" / "Couldn't load shared because it is already opened from another project or workspace." Cause: both apps reference the same local `shared` package, and Xcode only allows a local SwiftPM package open in **one** project/workspace at a time. Fix: keep only one app project open in Xcode at a time. (CLI builds are unaffected.)

Process:

- Introduced **milestone-based versioning** — `RELEASES.md` + annotated semver pre-1.0 tags (`v0.MINOR.0`) at verified, CI-green checkpoints only; convention recorded in `CLAUDE.md`. Tagged **`v0.1.0 — Authentication`** on a confirmed-green commit.
- Confirmed `project-manager-log.md` is the only once-a-day artifact (this journal); everything else follows normal commit cadence.

**Progress.** Authentication is functionally complete and verified end-to-end on both apps, CI is green, and we have a clean rollback point. The only Phase 1 work left is pre-auth UI and the account-state model.

**Steps for tomorrow.**

- Tag **`v0.2.0 — Onboarding`** once those are built and CI-green.
- Decide the **account state model** (logged out / no profile / pending / live / rejected) — likely fold into Phase 2/3 since the profile states don't exist yet.

Output a readable doc (e.g. `docs/codebase-overview.md`) aimed at someone new to the project.
